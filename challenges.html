              # Challenges of this Project

              # Problems Turning Requirements into Features

              There were two key problems with creating the feature list for development,

              • The requirements given by the client we detailed, but not framed in terms of our application model.

              • There was no clear lexicon in the problem description, so we had to establish terms to assist in communicating ideas, for example activity, gap, leave pool, advanced leave.

              # Algorithm challenges

              The largest piece of development was the leave calculation algorithm, which presented these problems,

              # No periodic finalization

              The concept of finalization is often used in leave management (and payroll) systems to simplify calculations. This distinguishes current period and historical data, and means that only current period data can be changed.

              It also places constraints on the user's process, making retrospective changes cumbersome and error prone. The client required that all records for all time should be changeable, and the leave calculation should recalculate all affected records.

              # Performance wrt retrospective changes

              Given the above requirement, a problem arises with performance. To get a fast calculation we needed to restrict data to the smallest set possible. Finding the calculation window was a challenge due to the Advance Leave requirement (where leave can be used before it is earned).

              # Advanced Leave

              Advanced Leave also made it harder to work out the record processing order, i.e sequential iteration does not work.

              # French legislation

              Legislation required all months to be “leveled” to 30 days, which meant a series of tweaks to leave earned and burned on the 31st of the month (with variations for February).

              # Handling data inconsistency

              Looking at the posting data already in the database, it was apparent that some of the records represented impossible situations, e.g a seafarer posted to two vessels at the same time.

              This occurred because multiple candidate postings are entered during the rostering process, with only one posting being implemented. The unused postings were often not discarded, making after-the-fact processing such as leave calculation difficult.

              # Incremental changes

              Requirements were added to the list as development progressed. Partly, this was because the client was building a payroll extractor at the same time and was discovering new requirements for the leave data, which fed into the payroll system.

              # Thirty Day Rule - French statutory requirement

              The special calculation step known as the Thirty Day Rule was tricky as it was defined from statutory legislation (so needed some interpretation) and the impact on the model were not well understood. This feature underwent some incremental changes during development.

              # Finance requirements to fully cost Seafarer’s pay

              Initially the leave earned was simply based on shifts worked, however many seafarers (particularly officers) are paid a standard salary regardless of time spent on a vessel, but are still required to follow the leave earned / leave burned model. This lead to the concept of Advanced and Anticipated vacation to fully cost these salaries.

              # Handling new (unspecified) activities that may be defined in the future

              The client wanted new activity types to be handled by configuration options rather than application changes. This was challenging, as up to this point we had been hard-coding activity variations.

              # Conforming to the Rostering application model

              One hard limitation was the core product could not be changed in any way. A certain amount of customization is available in MAPS Rostering, in the form of soft fields in the database and VBA hooks into the UI.

              These were used to integrate the Leave Module UI seamlessly from the user perspective, but required the Leave application to run with it’s own model and use adapters to reshape data moving out and writing back to the core database.

              # Model Adapter (Extractor)

              The Extractor pulls source records out of the database and transforms them into a format suitable for the allocation step.

              • Core Posting, Sickness, and Training records are mapped to an Activity type which is the base input record for allocation of leave.

              • Transits (travel days to and from a Posting or a Training Course) are attributes of the parent record. These are also mapped as activities so that rates can vary independently of the parent activity.

              • Sickness periods which span posted time and leave time are treated differently. On-board, the sickness period is effectively ignored. Ashore, sickness can use up leave entitlement so these records need to be split in the activity list.

              • Other reasons to split source records include award rate changes and personal change of leave group (benefit level).

              • Location (OnBoard, Ashore, Home) is not a native concept to the MAPS application, but is required for leave calculation as the rates applied can be different. This requires traversal of the organization tree to the root branch which represents ships and offices.
