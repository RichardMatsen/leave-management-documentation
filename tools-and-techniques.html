      # Tools and Techniques

      # Test Runner Application

      The test runner uses XML files of test data (both fixture and expected results) to provide automated acceptance testing which can be used by non-developer team members (including client).

      It loads the fixture data into a test database, invokes the leave calculation engine and checks the results, which are expressed as a simple SQL select statement.

      # Data Context Interaction (DCI)

      DCI is a paradigm whereby the code for a use-case is focused in a single module (context plus traits), as opposed to having many interacting methods on the domain objects.

      There are a large number of domain objects - contracts, activities, leave groups, rates, gaps, leave pool, diary, intervals. Having the algorithm fragmented between methods on these objects makes it hard to see the algorithm as a coherent whole.

      DCI solves this by moving use-case specific code out of the domain objects and into traits, which are entity extensions specific to the use-case.

      # Debugging and Support

      # Timeline Visualization

      When debugging a leave calculation, there is a lot of data to go through (calculations are made at the day level and can span years).

      To make it easier to pinpoint problems, a WPF application was built which gives a Gantt type display of the various dated records.

      # Calculation Data Extract

      To assist helpdesk support an Analysis Data Extract feature was provided. This serializes (to XML) the complete internal calculation model of a leave calculation run for a particular Seafarer.

      If the client perceives a problem with a calculation, this file could be emailed to Allocate Software Support and then loaded into a test case fixture for evaluation in the Test Runner application.
